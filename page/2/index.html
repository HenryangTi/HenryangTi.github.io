One of the most important capabilities in Spark is persisting (or caching) a dataset in memory across operations.When you persist an RDD, each node stores any partitions of it that it computes in memory and reuses them in other actions on that dataset (or datasets derived from it).This allows future actions to be much faster (often by more than 10x) Spark中最重要的能力是缓存一个在内存中的数据集，该数据集可被很多操作使用。当你持久化一个RDD时, 每个节点存储RDD的任意分区，这样在内存中RDD可以进行计算或者作用于该数据集上的其他action中重新使用被cache的RDD。这个允许未来的action速度更快(经常差不多是原来的10倍速度)

5.几种默认存储级别的官网描述

Storage Level Meaning Meaning的中文翻译
MEMORY_ONLY Store RDD as deserialized Java objects in the JVM. If the RDD does not fit in memory, some partitions will not be cached and will be recomputed on the fly each time they’re needed. This is the default level. 在JVM中以不序列化的Java对象存储RDD。如果RDD在not fit in 内存(也就是说该RDD中的partition的数据存储不了)，一些partition将不会被cache住, 并且将会在每次需要的时候重新计算。这是默认的级别。
Storage Level Meaning Meaning的中文翻译
MEMORY_AND_DISK Store RDD as deserialized Java objects in the JVM. If the RDD does not fit in memory, store the partitions that don’t fit on disk, and read them from there when they’re needed. 在JVM中以不序列化的Java对象存储RDD。如果RDD不能fit in 内存, 在disk上存储不能fit的partition，实用的时候再读出来。
Storage Level Meaning Meaning的中文翻译
MEMORY_ONLY_SER Store RDD as serialized Java objects (one byte array per partition). This is generally more space-efficient than deserialized objects, especially when using a fast serializer, but more CPU-intensive to read. 以序列化的方式存储RDD(每个partition一个字节数组)。该方式比不序列化的对象更加节省空间, 特别是使用快速的serializer, 但是读取的时候增加对CPU性能的消耗。

7.如何选择Storage Level

官网原文 中文翻译
Spark’s storage levels are meant to provide different trade-offs between memory usage and CPU efficiency Spark的存储级别的选择意味着是在内存使用和CPU效率之间做一个平衡
官网原文 中文翻译
We recommend going through the following process to select one：If your RDDs fit comfortably with the default storage level (MEMORY_ONLY), leave them that way.This is the most CPU-efficient option, allowing operations on the RDDs to run as fast as possible.If not, try using MEMORY_ONLY_SER and selecting a fast serialization library to make the objects much more space-efficient, but still reasonably fast to access. (Java and Scala).Don’t spill to disk unless the functions that computed your datasets are expensive, or they filter a large amount of the data.Otherwise, recomputing a partition may be as fast as reading it from disk.Use the replicated storage levels if you want fast fault recovery (e.g. if using Spark to serve requests from a web application).All the storage levels provide full fault tolerance by recomputing lost data, but the replicated ones let you continue running tasks on the RDD without waiting to recompute a lost partition. 我们推荐选择的时候采取如下步骤：如果你的RDD能够很好的适用默认的存储级别(MEMORY_ONLY), 就不需要改变。这是最省CPU的选择, 允许RDD上的操作运行的尽可能快。如果第一种(默认的)不行, 尝试使用MEMORY_ONLY_SER模式, 并且使用一个快速的序列化库, 这样可以使对象更加节省空间, 但是访问速度还是可以的。不要将RDD存储到磁盘上, 除非从数据集计算出RDD的函数是花费巨大的, 或者该函数过滤了大数据。重新计算一个分区可能比从磁盘中读RDD速度更快。如果你想快速容错, 可以使用副本的存储级别。所有的存储级别通过丢失数据后重新计算的方式提供了一个完整的容错机制, 但是副本的方式可以让你继续在RDD上运行task, 不需要等待重新计算丢失的partition

Spark Shuffle

2019-01-05

1.shuffle介绍

官网原文 中文翻译
Certain operations within Spark trigger an event known as the shuffle. The shuffle is Spark’s mechanism for re-distributing data so that it’s grouped differently across partitions. This typically involves copying data across executors and machines, making the shuffle a complex and costly operation. Spark的某些操作会触发一个事件，该事件就是shuffle。shuffle是spark中重新分发数据的机制，以便不同partition的数据都能分在一起。这类操作在多个executor或机器上复制数据，这样使shuffle是一个复杂的并且是耗费高的操作

2.shuffle图示

3.背景

官网原文 中文翻译
To understand what happens during the shuffle we can consider the example of the reduceByKey operation. The reduceByKey operation generates a new RDD where all values for a single key are combined into a tuple - the key and the result of executing a reduce function against all values associated with that key. The challenge is that not all values for a single key necessarily reside on the same partition, or even the same machine, but they must be co-located to compute the result. 为了理解当shuffle时候发生来什么，我们可以考虑一下reduceByKey。reduceByKey操作产生来一个新的RDD，在该RDD中属于一个key的所有的值都会被放入一个tuple中，最后结果是key与和该key相关的value进行reduce操作之后的值。但是这里的困难是对于一个key的所有value都存在与一个partition，甚至一个机器上，但是它们必须放在一起才能计算出结果

注意

官网原文 中文翻译
In Spark, data is generally not distributed across partitions to be in the necessary place for a specific operation. During computations, a single task will operate on a single partition - thus, to organize all the data for a single reduceByKey reduce task to execute, Spark needs to perform an all-to-all operation. It must read from all partitions to find all the values for all keys, and then bring together values across partitions to compute the final result for each key - this is called the shuffle. 在Spark中，数据一般不是跨分区分布的以便对于特定的操作，数据都在必要的位置。在计算中，单独的task会在单独的partition上进行操作，因此，为了管理数据(单独reduceByKey的reduce操作要执行的数据)，Spark需要来操作一个all-to-all的操作，它必须从所有的partition中读取数据来找到全部的值，将这些值跨分区的放在一起来对每个key进行计算-这种操作就叫做shuffle
官网原文 中文翻译
Although the set of elements in each partition of newly shuffled data will be deterministic, and so is the ordering of partitions themselves, the ordering of these elements is not. 尽管每个分区上最新shuffle过的数据是确定的，各个分区间的顺序是确定的，但是分区内元素的顺序是不确定的。

4.性能影响

官网原文 中文翻译
The Shuffle is an expensive operation since it involves disk I/O, data serialization, and network I/O. To organize data for the shuffle, Spark generates sets of tasks - map tasks to organize the data, and a set of reduce tasks to aggregate it. This nomenclature comes from MapReduce and does not directly relate to Spark’s map and reduce operations. Shuffle操作是一个耗费大的操作，因为它包括了磁盘I/O、数据序列化和网络I/O。为了组织shuffle操作所需要的数据，Spark产生了很多组tasks，map task来组织数据，一组reducetask来聚合数据。这种命名法来自于MapReduce，并且与Spark的map和reduce没有直接的关联

注

官网原文 中文翻译
Internally, results from individual map tasks are kept in memory until they can’t fit. Then, these are sorted based on the target partition and written to a single file. On the reduce side, tasks read the relevant sorted blocks. 内部上，单独map task的结果会保存在内存中直到内存不能再保存。然后，基于目标partition上排序后写到一个单个文件中。再reduce端，task会读取相关的排序的blocks
官网原文 中文翻译
Certain shuffle operations can consume significant amounts of heap memory since they employ in-memory data structures to organize records before or after transferring them. Specifically, reduceByKey and aggregateByKey create these structures on the map side, and ‘ByKey operations generate these on the reduce side. When data does not fit in memory Spark will spill these tables to disk, incurring the additional overhead of disk I/O and increased garbage collection. 某些shuffle操作能消费大量的堆空间因为他们应用了基于内存的数据结构来管理数据。详细的来说，reduceByKey和aggregateByKey在map端创建这些结构，然后ByKey类型的操作在reduce端生成这种数据结构。当数据不能再fit in内存的时候，Spark将会将这些table spill到磁盘上，给磁盘IO增加一些额外的开销并且增加垃圾回收次数
官网原文 中文翻译
Shuffle also generates a large number of intermediate files on disk. As of Spark 1.3, these files are preserved until the corresponding RDDs are no longer used and are garbage collected. This is done so the shuffle files don’t need to be re-created if the lineage is re-computed. Garbage collection may happen only after a long period of time, if the application retains references to these RDDs or if GC does not kick in frequently. This means that long-running Spark jobs may consume a large amount of disk space. The temporary storage directory is specified by the spark.local.dir configuration parameter when configuring the Spark context. Shuffle在disk上同样产生了大量的中间文件。在Spark1.3时, 这些中间文件被保存下来直到与之相关的RDD不再被使用或者已经被进行垃圾回收。这样如果当重新计算血缘关系的时候，这些shuffle的文件就不需要重新被创建。只有在运行很长时间之后才会进行垃圾回收，如果应用仍旧指向这些RDD或者GC不是经常生效。这就意味着长时间运行的Spark job会消耗大量的磁盘空间。该临时存储目录当配置Spark Context时被spark.local.dir参数所定义
